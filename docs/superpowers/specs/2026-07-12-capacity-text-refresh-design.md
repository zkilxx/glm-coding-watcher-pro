# “购买人数过多”刷新逻辑设计

## 目标

页面出现“购买人数过多”文字时，仍优先执行正常套餐检查和自动点击；只有没有可购买目标时才刷新，避免提示文字阻断购买命中。

## 运行逻辑

1. 从页面可见文本中检测“购买人数过多”，不要求文字位于弹窗中。
2. 不查找、不关闭任何提示框。
3. 套餐区域已加载时，按现有优先级完整检查目标套餐。
4. 发现可购买目标时，继续执行后台单次点击锁定、自动点击一次并停止刷新。
5. 所有目标不可购买时，记录刷新次数并立即刷新。
6. 如果出现“购买人数过多”且套餐区域尚未加载，也记录刷新次数并立即刷新，不停留在“等待套餐加载”。
7. 如果未出现该文字且套餐区域尚未加载，继续等待 DOM 渲染，避免普通加载过程误刷新。

## 安全边界

- 不绕过登录、验证码、限流或风控。
- 不调用隐藏购买接口。
- 不自动确认订单或支付。

## 测试

- 验证检测范围是页面文字而不是弹窗选择器。
- 验证人数过多且套餐未加载时调用刷新。
- 验证人数过多且存在可购买目标时仍进入现有点击流程。
- 验证普通页面未加载时仍等待，不误刷新。
