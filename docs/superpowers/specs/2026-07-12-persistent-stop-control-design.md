# 持久停止控制与弹窗按钮布局设计

## 目标

解决 GLM 页面自动刷新后 Chrome 扩展弹窗关闭、用户难以及时停止监测的问题，并修正弹窗底部操作按钮未对齐的问题。

## 交互设计

- 监测运行时，在 GLM 页面右上角显示一个固定悬浮信息状态卡。
- 状态卡显示绿色运行状态、“GLM 监测运行中”、本轮真实刷新次数、DOM 检测间隔和整行“停止监测”按钮。
- 每次点击“开始监测”时刷新次数归零；每次实际调用 `location.reload()` 前刷新次数加 1 并保存到后台。
- 页面刷新后，内容脚本从后台读取运行状态、刷新次数和检测间隔并重新创建状态卡。
- 点击网页中的“停止”后，向后台发送 `STOP_MONITORING`，停止检测与刷新，并立即移除控制条。
- 未运行、已经成功点击或用户停止后，不显示控制条。
- 状态卡使用 Shadow DOM 隔离样式，避免受 GLM 页面 CSS 影响，也不修改页面业务内容。
- “监测设置”改为纵向信息结构：顶部显示运行参数，DOM 检测间隔独占一行，声音提醒和桌面通知在独立开关区域，开始与停止按钮在底部等宽对齐。

## 数据流

1. 弹窗启动监测，后台保存标签页运行状态并通知内容脚本。
2. 内容脚本进入运行状态时创建悬浮状态卡。
3. 内容脚本刷新前发送 `REFRESH_RECORDED`，后台增加 `refreshCount`，随后执行刷新。
4. 页面刷新后内容脚本重新注入，通过 `GET_STATE` 恢复运行状态、刷新次数和状态卡。
5. 用户点击悬浮“停止”，内容脚本发送包含当前标签页来源的 `STOP_MONITORING`。
6. 后台将运行状态设为停止并回传；内容脚本清理定时器、观察器和状态卡。

## 错误处理

- 停止消息失败时，按钮恢复可点击并显示“重试停止”，避免假装已经停止。
- 重复点击停止按钮不会创建多个请求。
- 状态卡固定在右上角并使用高层级，宽度保持紧凑，不遮挡订阅卡片主体区域。
- 不处理验证码、登录、限流、风控、订单确认或支付。

## 测试

- 自动测试验证运行状态会创建状态卡，停止状态会移除状态卡。
- 自动测试验证网页停止按钮发送 `STOP_MONITORING`。
- 自动测试验证开始时 `refreshCount` 为 0，每次 `REFRESH_RECORDED` 原子增加 1。
- 自动测试验证刷新恢复路径仍调用 `GET_STATE` 并重建包含最新次数的状态卡。
- 样式测试验证监测设置分区清晰、弹窗操作区为等宽两列、按钮尺寸一致。
- 在 Chrome 中验证：启动监测、页面刷新、状态卡计数增加并重新出现、网页按钮停止、监测设置与操作按钮对齐。
