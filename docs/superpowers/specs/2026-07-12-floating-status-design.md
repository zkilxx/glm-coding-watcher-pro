# 悬浮窗实时状态设计

## 目标

网页右上角悬浮窗除刷新次数和停止按钮外，明确显示当前监测状态，让用户能区分抢购拥堵、套餐售罄、正在检查和已经命中。

## 状态与优先级

页面每次检查时按以下优先级确定状态：

1. 找到可购买的目标套餐：`发现可购买套餐`，随后执行单次点击并停止刷新。
2. 页面出现“抢购人数过多，请刷新再试”或“购买人数过多”：`抢购人数过多`。
3. 目标套餐按钮包含“暂时售罄”或“售罄”：`暂时售罄`。
4. 正在遍历套餐优先级：`检查套餐中`。
5. 点击完成：`已自动点击，刷新已停止`。

若同一页面同时出现拥堵文案和售罄按钮，以“抢购人数过多”为页面状态；但仍执行完整的目标套餐检查，可购买目标的优先级最高。

## 数据流

内容脚本从当前页面可见文本和目标套餐按钮计算状态，通过现有 `RUNTIME_STATUS` 消息写入标签页运行记录。悬浮窗渲染该运行状态，并在每次检查时更新文字。自动刷新后，内容脚本重新挂载悬浮窗，并从后台运行记录恢复状态和刷新次数。

## 界面

悬浮窗标题保持“GLM 监测运行中”。指标区调整为：

- 当前状态：占整行，显示上述状态文字。
- 刷新次数。
- 检测间隔。
- 停止监测按钮。

状态文字使用紧凑徽标样式，不增加新的操作入口。

## 边界

- 不改变套餐优先级、刷新、单次点击和停止逻辑。
- 不关闭提示框，不绕过验证码、登录、限流、风控或支付确认。
- 未知页面状态不伪装为售罄，显示“检查套餐中”。

## 验证

- 自动测试覆盖拥堵、售罄、检查中、命中和刷新恢复。
- 在真实 GLM 页面验证当前“暂时售罄”文案能显示为“暂时售罄”。
- 完整测试、JavaScript 语法检查和差异格式检查全部通过后发布。
